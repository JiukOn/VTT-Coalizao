================================================================================
          ANALYSIS AND INITIALIZATION — VTT COALIZAO PROJECT
================================================================================

1. OVERVIEW AND HOW TO START THE PROJECT
--------------------------------------------------------------------------------
VTT Coalizao is a complete Virtual Tabletop created with React/Vite on the frontend and Node.js/Express/WebSocket on the backend.

As the Game Master (Local Host):
- You can use the `StartDev.bat /s` script in the root of the project. It will verify, install dependencies if necessary, and start both the Frontend (Vite) and the Local Server (Node.js on port 3001).
- Access `http://localhost:5173` to open the GM Interface.

As a Player (Local/VPN Access):
- The GM provides their IPv4 Local IP (e.g., 192.168.0.15) and the Session Code.
- Players access via browser: `http://192.168.0.15:3001/#/player`.
- If players are outside the GM's network, they will need to use a VPN (ZeroTier, Hamachi, Radmin VPN) to simulate a local network, then input the IP provided by the VPN.

2. WHAT SHOULD BE UPLOADED TO GITHUB
--------------------------------------------------------------------------------
You should keep the repository light and organized, sending Only the Source Code Files.
The `.gitignore` file should already handle most of this.

MANDATORY Files on Git (SOURCE CODE):
- `src/` (All React frontend and base JSON data)
- `server/` (Local host Node server)
- `relay/` (Stateless online server)
- `public/` (Static files and Service Worker)
- `Docs/` (Planning and text files, including this one)
- Configuration files: `package.json`, `vite.config.js`, `eslint.config.js`, `.gitignore`, `*.bat`.

DO NOT upload (Add to .gitignore if not present):
- `node_modules/` (Heavy dependencies)
- `dist/` (Built files, unless in the `gh-pages` branch for deployment)
- `.env` or secret API keys (if they exist in the relay, for example)
- Folders with user temporary saves or locally created databases.

3. WHAT STILL NEEDS TO BE DONE TO BE 100% FUNCTIONAL
--------------------------------------------------------------------------------
Even though Phases 1 through 9 in Plan.txt are marked as completed, practically speaking, for a fluid session, you still need to ensure:
1. Image/Token Synchronization: Since you use local WebSockets, if the GM uploads an image from their PC to the map or an entity (saved in IndexedDB), the players' computers won't magically get that image. It is necessary to serve a static route in Express (`server/index.js`) or convert images to lightweight Base64 to transfer via WebSocket.
2. Player State Persistence on the Relay: If using the online Relay, if the GM drops internet, the temporary state might be lost if the relay is 100% stateless. You need to ensure automatic reconnection while restoring the map state for the players.
3. JSON/Data Limit Testing: IndexedDB works wonderfully, but as the world grows, importing and exporting massive JSONs can freeze browser tabs.
4. Disconnection Validations: What happens to a player's sheet if they disconnect during attribute distribution or if they try to use an ability while offline?

4. IMPROVEMENTS TO THE SERVER START PROCESS AND PLAYER ACCESS
--------------------------------------------------------------------------------
Currently, the process requires the player to know the GM's IP, use port :3001, and access the network via LAN or VPN like ZeroTier/Hamachi. This is a bit intimidating for RPG players without networking knowledge.

Suggested Improvements:
A. INTEGRATED PUBLIC TUNNEL (Easy):
- Integrate technologies like `ngrok` or `localtunnel` directly into `server/index.js`.
- The Node server, when started by the GM, automatically generates a temporary tunnel (e.g., `https://coalizao-session-gm.loca.lt`).
- The GM simply sends this link to the players on Discord. No need for IPs, ports, or Hamachi.

B. PACKAGING VIA ELECTRON / TAURI (Medium):
- The GM shouldn't need to run commands like `StartDev.bat` or use the Command Prompt.
- Packaging the project (`React` + `Express` Node together in Electron), they just open an `.exe`. The application opens visually, the server spins up hidden in the background, and the access link is displayed on screen.

C. WEBRTC INTEGRATION (Advanced):
- Remove the need for Node/Express as a relay and focus on WebRTC.
- The game would connect players via a signaling API (Basic Relay) and then they share data P2P (Peer-to-Peer), which reduces chances of network server lag and fixes most NAT and Firewall problems that block visibility between computers without Hamachi.

CONCLUSION:
The project is very well-founded. The main barrier between a "functional tech demo" and a "polished RPG product" will be facilitating network connections for laypeople (Public Tunnels instead of IP/VPN) and ensuring that binary files/images transferred by the Host appear efficiently on user screens.
