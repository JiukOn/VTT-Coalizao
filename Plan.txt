  1. VISÃO GERAL DO PROJETO
================================================================================

O VTT Coalizão é uma Mesa Virtual de RPG (Virtual Tabletop) projetada
exclusivamente para o sistema homebrew "Coalizão". Seu objetivo é fornecer
ao Mestre todas as ferramentas para conduzir sessões completas de RPG e,
progressivamente, permitir que os jogadores participem remotamente com suas
próprias visões da mesa.

  MODO MESTRE (Host) — Controle total, execução local:
  • Gerenciamento completo de fichas de heróis, NPCs e criaturas
  • Rolagem de dados restrita ao sistema (apenas D20 e D4)
  • Mapas interativos com grid, tokens, névoa, linhas e visibilidade
  • Banco de dados integrado (habilidades, armas, itens, modificações)
  • Tracker de combate com iniciativa, efeitos e condições
  • Exportação/Importação total da campanha em JSON
  • Servidor local que aceita conexões de jogadores (LAN/VPN)

  MODO JOGADOR (Client) — Acesso limitado, conectado ao Host:
  • Visão do mapa com névoa de guerra ativa (apenas campo visível)
  • Consulta e edição da própria ficha de personagem
  • Rolagem de dados vinculada ao próprio personagem
  • Ver iniciativa atual e lista de turno
  • Resolução de combate para o próprio personagem
  • Consulta de habilidades e itens próprios
  • Notas e diário de sessão (salvo como JSON no Host)
  • Sem acesso a entidades, mapas globais ou dados de outros jogadores

  PRINCÍPIOS DO PROJETO:
  • OFFLINE-FIRST: funciona completamente sem internet (Host local)
  • PROGRESSIVO: começa simples (local), evolui para online
  • SEGURO: Host controla o que cada jogador pode ver e fazer
  • LEVE: < 2MB build estático sem imagens do usuário

O programa deve ser LEVE, BONITO, COMPLETO e FÁCIL DE DISTRIBUIR.

  2. ARQUITETURA DO SISTEMA (REVISADO v4.0)
================================================================================

  ┌─────────────────────────────────────────────────────────────────────┐
  │  EVOLUÇÃO DA ARQUITETURA                                           │
  │                                                                    │
  │  Fase 1-6: SPA puro (React + Vite), sem servidor, tudo local       │
  │  Fase 7A:  SPA + servidor local (Express/Node), LAN/VPN            │
  │  Fase 7B:  SPA + servidor remoto (WebSocket), conexão online       │
  └─────────────────────────────────────────────────────────────────────┘

  FASE 1-6 — STACK ATUAL:
  ─────────────────────────────────────────────────────────────────────
  FRAMEWORK:       Vite 8 + React 19
  ESTILIZAÇÃO:     Vanilla CSS com Design System customizado
  ARMAZENAMENTO:   IndexedDB (via Dexie.js 4) + localStorage
  EXPORTAÇÃO:      JSON (campanhas, personagens, mapas)
  IMAGENS:         Base64/Blob no IndexedDB
  ÍCONES:          Lucide Icons
  FONTES:          Inter (UI), Cinzel (títulos), JetBrains Mono (stats)
  CANVAS:          HTML5 Canvas API para mapa e grid

  FASE 7A — SERVIDOR LOCAL (LAN/VPN):
  ─────────────────────────────────────────────────────────────────────
  SERVIDOR:        Node.js + Express (backend leve)
  COMUNICAÇÃO:     WebSocket (ws ou socket.io) para sincronização
  AUTENTICAÇÃO:    Código de sessão gerado pelo Mestre (6 dígitos alfanum.)
  ARMAZENAMENTO:   JSON files no disco do Host para saves de jogadores
  REDE:            LAN local ou VPN (ex: Hamachi, ZeroTier, Tailscale)
  HOSPEDAGEM:      GitHub Pages para página de login/lobby

  COMO FUNCIONA (7A):
  1. Mestre abre o VTT e inicia um servidor local (botão "Iniciar Servidor")
  2. VTT exibe o IP local da máquina e um código de sessão (ex: "GH4K9X")
  3. Jogadores acessam: https://[usuario].github.io/VTT-coalizao
  4. Digitam o IP do Host + código da sessão para conectar
  5. Após autenticar, recebem a visão de jogador com seu personagem
  6. Sincronização em tempo real via WebSocket

  FASE 7B — CONECTIVIDADE ONLINE:
  ─────────────────────────────────────────────────────────────────────
  SERVIDOR REMOTO: Node.js em VPS (Railway, Render, Fly.io — tier gratuito)
  BANCO REMOTO:    PostgreSQL ou SQLite no servidor (apenas metadados de sessão)
  DADOS PESADOS:   Permanecem no IndexedDB local do Mestre
  SINCRONIZAÇÃO:   Deltas (apenas mudanças) via WebSocket para economizar banda
  CONEXÃO:         HTTPS + WSS (WebSocket Secure)
  LOGIN:           Username + código de campanha (sem senha = simplicidade)

  DOIS PROGRAMAS (MODOS):
  ─────────────────────────────────────────────────────────────────────
  MODO MESTRE:
  • Acesso a todos os módulos e telas
  • Pode criar, editar e deletar qualquer entidade
  • Controla névoa de guerra, visibilidade e estado do mapa
  • Gerencia o servidor e as conexões dos jogadores
  • Vê tudo: fichas de todos, posições, efeitos, HP

  MODO JOGADOR:
  • Acesso apenas à própria ficha e ao mapa (com névoa)
  • Pode editar anotações, inventário e status da própria ficha
  • Não pode ver dados de outros jogadores
  • Não pode mover tokens que não sejam os seus
  • Pode rolar dados e consultar suas habilidades/itens

  ORDEM DE IMPLEMENTAÇÃO:
  1. Completar o Modo Mestre (Fases 1-6) como SPA standalone
  2. Criar o Modo Jogador como sub-rota/visão do mesmo app
  3. Adicionar servidor local (Fase 7A) para conectar os dois
  4. Evoluir para conectividade online (Fase 7B)

  • Build estático via "npm run build" → pasta /dist
  • GitHub Pages: https://[usuario].github.io/VTT-coalizao

  • Frontend continua no GitHub Pages
  • Backend deployado em serviço gratuito (Railway, Render ou Fly.io)
  • Conexão via HTTPS/WSS sem necessidade de VPN
  • Limite: manter o tier gratuito (512MB RAM, ~500 conexões simultâneas)

  CONFIGURAÇÃO DO VITE:
  • vite.config.ts → base: '/VTT-coalizao/'
  • Manifest.json para PWA
  • Service Worker para cache offline

  5.2  SISTEMA DE ROLAGEM DE DADOS (DICE ROLLER)
────────────────────────────────────────────────────────────────────────────────

  REGRA FUNDAMENTAL: Sistema Coalizão usa APENAS D20 e D4.

  CLASSIFICAÇÃO AUTOMÁTICA:
  Para D20: 20=Crítico | 13-19=Bom | 10-12=Normal | 2-9=Ruim | 1=Desastre
  Para D4:  4=Melhor | 3=Bom | 2=Ruim | 1=Pior

  FUNCIONALIDADES:
  • Modificador de bônus (+X)
  • Vantagem (rola 2, pega maior) e Desvantagem (rola 2, pega menor)
  • Quantidade de dados (1 a 10)
  • Histórico scrollável com timestamp e classificação
  • Som opcional (dado caindo na mesa)
  • Vinculação a personagem/atributo para bônus automático

  5.3  GERENCIADOR DE PERSONAGENS — FICHAS COMPLETAS (REVISADO v4.0)

  ── ESPÉCIES (REVISADO) ────────────────────────────────────────────────────
  Cada personagem tem ESPÉCIE PRIMÁRIA (obrigatória) e ESPÉCIE SECUNDÁRIA
  (opcional, exceto se a primária for Anjo).

  ESPÉCIES PRIMÁRIAS disponíveis:
  ┌──────────────────────────────────────────────────────────────────────┐
  │ Humano     │ Padrão, sem modificadores especiais                     │
  │ Elfo       │ Gracioso, bônus DEX/INT                                 │
  │ Anão       │ Resistente, bônus VIT/FRC                               │
  │ Gigante    │ Forte, bônus FRC/VIT, penalidade DEX                    │
  │ Kobold     │ Ágil e furtivo, bônus DEX/CRM                           │
  │ Goblin     │ Esperto e caótico, bônus INT/DEX                        │
  │ Yomunkai   │ Disciplinado, bônus RES/PRE                             │
  │ Demônio    │ Poderoso, bônus ENR/FRC, penalidade CRM                 │
  │ Anjo       │ Divino, bônus CRM/ENR — PROIBE espécie secundária       │
  └──────────────────────────────────────────────────────────────────────┘

  ESPÉCIES SECUNDÁRIAS disponíveis:
  ┌──────────────────────────────────────────────────────────────────────┐
  │ Nenhuma    │ Personagem de espécie pura                              │
  │ [Qualquer  │ Qualquer uma das espécies primárias (exceto Anjo)       │
  │  primária] │                                                          │
  │ Planta     │ Regeneração passiva, fraqueza ao fogo                   │
  │ Gran       │ Grande porte físico, bônus VIT                          │
  │ Monstro    │ Traços bestiais, habilidade natural de ataque           │
  │ Bom        │ Alinhamento puro, bônus CRM e resistência mágica        │
  └──────────────────────────────────────────────────────────────────────┘

  Regras de combinação:
  • Anjo como primária → campo secundário desabilitado automaticamente
  • Duas espécies iguais não são permitidas
  • Efeitos das duas espécies se somam (bônus e penalidades)
  • Interface: dois selects sequenciais, o segundo atualiza baseado no primeiro

  ── ALINHAMENTOS (REVISADO) ────────────────────────────────────────────────
  Alinhamentos são PERSONALIZADOS pelo Mestre e pelo jogador:
  • Não há lista predefinida — qualquer texto é válido
  • Campo de entrada de texto livre para adicionar alinhamentos
  • Personagem pode ter múltiplos alinhamentos
  • Interface: lista expandível com botão "Adicionar Alinhamento"
    - Clicar abre um campo de texto para digitar o novo alinhamento
    - Lista exibe todos os alinhamentos com botão de remoção
    - Exemplos livres: "Leal ao Rei", "Vendetta contra os Elfos",
      "Protetor de Crianças", "Caçador de Artefatos"

  ── TENDÊNCIAS ──────────────────────────────────────────────────────────────
  Tendências concedem DADO DE VANTAGEM em testes relacionados:
  • Até 3 tendências por personagem
  • Seleção de lista configurável (Mestre pode adicionar/remover)
  • Exemplos: Botânica, Estratégia, Forja, Alquimia, Culinária, Música,

  ── CLASSE E HABILIDADES INICIAIS (REVISADO) ──────────────────────────────
  SELEÇÃO DE CLASSE:
  • Interface de seleção com visualização de multiplicadores
  • Para cada classe exibir:
    - Multiplicadores de atributo (ex: FRC ×1.5, DEX ×1.2)
    - Valores FINAIS previstos com base nos pontos distribuídos
    - Bônus previstos (floor(valorFinal / 5))
    - Habilidade Legado vinculada à classe
  • Opção especial "Não Classificado":
    - ENR não pode ser aumentado (fica em 0)
    - Não é possível selecionar habilidades
    - Multiplicadores: todos ×1.0

  HABILIDADES INICIAIS:
  • Cada personagem começa com 2 habilidades (vinculadas à classe)
  • Habilidade 1: automática (habilidade Legado da classe)
  • Habilidade 2: SELECIONÁVEL pelo jogador entre as opções da classe
  • Interface: dropdown com as opções disponíveis para a classe escolhida
  • Se "Não Classificado": nenhuma habilidade disponível

  │ FRC (Força)      │                                                  │
  │ INT (Inteligência│                                                  │
  │ RES (Resiliência)│                                                  │
  │ PRE (Precisão)   │                                                  │

  • Total de pontos iniciais: 25 (validação automática)
  • Multiplicador definido pela classe
  • Ao subir de nível, pontos extras conforme tabela
  • Interface mostra: pontos gastos / pontos disponíveis em tempo real

  5.4  GERENCIADOR DE ENTIDADES (NPCs, Criaturas, Aliados)
────────────────────────────────────────────────────────────────────────────────

  TIPOS: Herói | NPC | Criatura | Criatura Especial | Aliado

  FICHA DE CRIATURA:
  • Nome, imagem/token, tipo (terrestre/aquático/voador/semi-aquático)
  • Dieta, tamanho (pequeno/médio/grande/colossal), elemento
  • Atributos simplificados (VIT, DEX, FRC, PRE, ENR, RES, INT, CRM)

  • Notas do Mestre (campo privado)
  • Relação com heróis (Amigável/Neutro/Hostil)

  FUNÇÕES:

  • Cada personagem/token na mesa pode ter um CÍRCULO DE VISIBILIDADE
  • Raio do círculo configurável (em quadrados de grid)
  • O círculo automaticamente revela células dentro do raio
  • Configuração por token: botão direito → "Configurar Visibilidade"

  VISÃO DIRECIONAL (CONE DE VISÃO) — NOVO:
  • Cada token pode ter uma DIREÇÃO DE VISÃO (ângulo configurável)
  • A visão na direção frontal tem alcance 1.2× o raio do círculo
  • A abertura do cone é um funil: ~120° de amplitude total
  • Visual: círculo base + cone brilhante na frente
  • Jogador pode rotacionar a visão do seu token pelo mouse ou teclado

  BLOQUEIO DE LINHA DE VISÃO (NOVO):
  • Linhas retas/curvas desenhadas no mapa funcionam como PAREDES
  • O sistema traça raios do token em direção às células
  • Se um raio atingir uma parede (linha desenhada), as células atrás
    desta parede ficam NA SOMBRA (ocultas mesmo dentro do raio)
  • Algoritmo: ray casting para cada célula visível
  • Resultado: tokens atrás de paredes não são visíveis pelo jogador
  • Paredes podem ser marcadas como "transparentes" (não bloqueiam visão)

  RESUMO DO SISTEMA DE VISIBILIDADE:
  1. Token A tem raio 5 e ângulo 0° (frente = leste)
  2. Sistema gera círculo de raio 5 ao redor do token
  3. Cone 1.2× (raio 6) na direção leste, abertura 120°
  4. Para cada célula no círculo/cone, traça raio do token
  5. Se raio cruzar uma parede → célula e tudo atrás ficam na sombra
  6. Células visíveis são reveladas automaticamente na névoa

  5.18 SERVIDOR E CONECTIVIDADE (NOVO v4.0)
────────────────────────────────────────────────────────────────────────────────

  FASE 7A — SERVIDOR LOCAL (LAN/VPN):
  ─────────────────────────────────────────────────────────────────────
  TECNOLOGIA: Node.js + Express + ws (WebSocket nativo)

  COMO O MESTRE INICIA O SERVIDOR:
  1. Mestre clica em "Iniciar Servidor" no painel de configurações
  2. VTT inicia servidor Node.js na porta 3000 (configurável)
  3. App exibe: IP local da máquina + código de sessão gerado
  4. Mestre compartilha o código com os jogadores (WhatsApp, Discord, etc.)
  5. Jogadores conectam via página no GitHub Pages

  FORMATO DO HANDSHAKE:
  • Jogador envia: { type: "join", playerName: "Akali", campaignCode: "GH4K9X" }
  • Host responde: { type: "welcome", playerId: "uuid", characterData: {...} }
  • Ou: { type: "error", message: "Código inválido" }

  EVENTOS EM TEMPO REAL (WebSocket):
  • map_update: Mestre move token / revela névoa / atualiza mapa
  • entity_update: HP, efeitos, iniciativa de uma entidade
  • combat_event: Resultado de combate para exibir no log do jogador
  • turn_change: Mudança de turno no tracker de iniciativa
  • dice_roll: Rolagem de algum jogador para o Mestre ver
  • notes_save: Notas do jogador chegando ao Host para salvar

  ARMAZENAMENTO NO HOST:
  • Saves de jogadores: /saves/{sessionCode}/{playerName}.json
  • Notes de sessão: /saves/{sessionCode}/notes/{playerName}_{session}.json
  • Backup automático a cada mudança de estado significativa

  ALTERNATIVA DE REDE (sem configuração de roteador):
  • Hamachi, ZeroTier ou Tailscale para VPN peer-to-peer
  • Instruções de instalação incluídas no README do projeto
  • Alternativa no futuro (Fase 7B): conexão direta pela internet

  FASE 7B — CONECTIVIDADE ONLINE:
  ─────────────────────────────────────────────────────────────────────
  SERVIDOR REMOTO: Node.js + Express + Socket.io
  HOSPEDAGEM: Railway / Render / Fly.io (tier gratuito)
  BANCO DE SESSÕES: SQLite ou PostgreSQL (apenas metadados leves)

  DADOS NA NUVEM (mínimo para economizar recursos):
  • Registro de sessões ativas (código, host IP, timestamp)
  • Metadados de jogadores conectados
  • Estado de sincronização (diff/deltas)

  DADOS PERMANENTES (sempre no Host local):
  • Campanha completa (personagens, criaturas, itens, habilidades)
  • Mapas e imagens
  • Histórico de sessões

  MODELO DE SINCRONIZAÇÃO:
  • Host → Servidor: registra sessão com código, recebe relay ID
  • Jogador → Servidor: conecta com código, server redireciona para Host
  • Servidor funciona como RELAY (ponte), não armazena dados da campanha
  • Conexão Host-Jogador é P2P via servidor (não centralizada)

  SEGURANÇA:
  • Código de sessão é único e expira após a sessão
  • Dados da campanha nunca saem do PC do Host
  • HTTPS/WSS obrigatório para todas as conexões

  5.19 ESTRUTURA DE DADOS UNIFICADA (JSON / I18N)

  Para facilitar a expansão, modding e internacionalização (I18N), TUDO no VTT
  deve ser armazenado em arquivos JSON individuais separados por pastas, com as
  informações duplicadas internamente nas chaves "pt-br" e "en-us".

  REQUISITOS POR ENTIDADE:
  • Personalidades: Nome, Descrição
  • Classes: Nome, Descrição, Habilidade Legado, Multiplicadores
  • Efeitos: Nome, Descrição, Tipo (ativo/passivo), Duração Temporária (turnos; usar -1 se permanente), Opções de Cura/Desativação, Categoria (psicológico, doença, ferida, etc.)
  • Auras: Nome, Descrição, Momentos de Ativação Forçada
  • Espécies: Nome, Descrição, Multiplicadores da espécie
  • Modificações: Nome, Descrição, Atributos modificados (positivo ou negativo)
  • Tendências: Nome, Descrição de Uso
  • Ambientes (Environments): Nome, Descrição
  • NPCs: Nome, Descrição, Relação/Tipo (Aliado, Neutro, Hostil, Comerciante, Nobre, Guarda, Outro),
           Personalidade (lista de referências a personalidades), Possível Benefício, Possível Malefício, Ficha
  • Criaturas: Nome, Descrição, Núcleo de Energia (pequeno +1, médio +2, grande +3, colossal +4),
               Tamanho (minúsculo, pequeno, médio, grande, colossal, mundial),
               Elementos (lista contendo referências aos elementos), Ficha completa (stats idênticos ao herói)
  • Elementos: Nome, Descrição (Banco de dados completo — fogo, água, terra, gelo, relâmpago, madeira,
               sombra, ar, luz, pedra, vazio)
  • Domínios: Nome, Descrição, Líder, Localização Principal, Tipo (Reino/Cidade/Academia/etc.), Alinhamento

================================================================================
  7. FLUXO DE USO PRINCIPAL (USER JOURNEY)
================================================================================

  MODO MESTRE — PREPARAÇÃO DA SESSÃO:
  ─────────────────────────────────────────────────────────────────────
  1. Mestre abre o VTT no navegador (localhost ou GitHub Pages)
  2. Seleciona ou cria uma campanha
  3. Cadastra/edita personagens dos jogadores (fichas completas)
  4. Prepara encontros: seleciona criaturas/NPCs do banco
  5. Cria ou importa mapas, configura o grid e as paredes (linhas)
  6. Posiciona tokens, configura visibilidade por personagem
  7. Se for sessão online (Fase 7): inicia o servidor, anota código e IP
  8. Anota lembretes no gerenciador de campanha

  MODO MESTRE — DURANTE A SESSÃO:
  ─────────────────────────────────────────────────────────────────────
  1. Narra e usa o mapa para mostrar cenários (revela névoa progressivamente)
  2. Gerencia visibilidade dos tokens (círculos e cones por personagem)
  3. Em combate: clica "Rolar Iniciativa" → ordem automática
  4. Usa CombatResolver para resolver ataques com bônus automáticos
  5. Aplica dano e efeitos; tracker decrementa efeitos a cada turno
  6. Avança turnos, checklist de ações reseta a cada novo turno
  7. Registra dano, cura e status em tempo real

  MODO JOGADOR — DURANTE A SESSÃO (Fase 7):
  ─────────────────────────────────────────────────────────────────────
  1. Acessa a página de login com código fornecido pelo Mestre
  2. Vê o mapa com névoa (apenas campo do seu token visível)
  3. Consulta sua ficha, habilidades e itens
  4. Rola dados, resolve seus ataques, move seu token
  5. Faz anotações de sessão (salvas automaticamente no Host)
  6. Acompanha o tracker de iniciativa para saber quando é seu turno

  PÓS-SESSÃO:
  ─────────────────────────────────────────────────────────────────────
  1. Mestre registra notas da sessão
  2. Atualiza XP e nível dos personagens
  3. Exporta backup da campanha em JSON
  4. Fecha o navegador — dados ficam salvos no IndexedDB

  8. REQUISITOS DE PERFORMANCE E PESO
================================================================================

  META DE TAMANHO DO BUILD (Fases 1-6, frontend):
  • Bundle JS: < 500KB (gzipped total)
  • CSS: < 50KB
  • Total estático: < 2MB (sem imagens do usuário)

  META DE PERFORMANCE:
  • First Contentful Paint: < 1.5s
  • Time to Interactive: < 2.5s
  • Canvas do mapa a 60fps (pan, zoom, tokens)
  • Drag de tokens sem lag perceptível
  • Ray casting de linha de visão: < 16ms por frame (memoização agressiva)

  META DO SERVIDOR (Fase 7):
  • Suporta até 8 jogadores simultâneos no mesmo servidor local
  • Latência WebSocket < 50ms na LAN
  • Save de notes do jogador: < 100ms
  • Servidor leve: < 50MB RAM em idle

  OTIMIZAÇÕES PLANEJADAS:
  • Lazy loading por aba (code splitting automático do Vite)
  • Memoização de ray casting (recomputar apenas quando mapa muda)
  • Canvas com double buffering para névoa e visibilidade
  • Debounce em inputs de busca (300ms)
  • React.memo em componentes pesados (Token, EntityCard, etc.)
  • Service Worker para cache offline (PWA, Fase 6)
  • Delta sync via WebSocket (não envia estado completo a cada mudança)

  [BUG-005] ✅ RESOLVIDO (13/04/2026) — Heróis e NPCs não apareciam nas páginas
  Causa: Tabelas 'npcs' e 'characters' no IndexedDB não tinham 'isCustom' como campo
         indexado. dataSeeder.js usava db.npcs.where('isCustom').equals(0).delete()

         impedindo que QUALQUER dado fosse semeado.
  Fix:   database.js v2: adicionado 'isCustom' nos índices de 'npcs' e 'characters'.
         dataSeeder.js: CURRENT_SEED_VERSION → '2.1.0' para forçar re-seed.

  12.7 PROTOCOLO WS ENTRE DOMÍNIOS (REFERÊNCIA)
  ─────────────────────────────────────────────────────────────────────────────

  As mensagens WebSocket são a única fronteira de comunicação entre
  USER-MASTER e USER-PLAYER. Cada mensagem carrega role para validação:

  MASTER → SERVER → PLAYER:
  ┌────────────────────┬──────────────────────────────────────────────┐
  │ type               │ payload                                      │
  ├────────────────────┼──────────────────────────────────────────────┤
  │ map_update         │ { mapData, fogCells, wallSegments }          │
  │ entity_update      │ { entities[] } (filtrado: sem dados privados)│
  │ initiative_update  │ { order[], currentTurn }                     │
  │ combat_event       │ { log entry, attacker, defender, result }    │
  │ broadcast_message  │ { text, sender: 'master' }                   │
  └────────────────────┴──────────────────────────────────────────────┘

  PLAYER → SERVER → MASTER:
  ┌────────────────────┬──────────────────────────────────────────────┐
  │ type               │ payload                                      │
  ├────────────────────┼──────────────────────────────────────────────┤
  │ join               │ { code, playerName, role: 'player' }         │
  │ join_character     │ { characterId }                              │
  │ player_roll        │ { result, type, context }                    │
  │ player_action      │ { action, characterId }                      │
  │ save_note          │ { text, timestamp }                          │
  │ ping               │ { timestamp } (keepalive 25s)                │
  └────────────────────┴──────────────────────────────────────────────┘
